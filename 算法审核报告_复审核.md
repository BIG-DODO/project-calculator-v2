## 注意事项（不影响功能，建议代码实现时留意）

### 1. `buildSupport` 注释笔误（极低优先级）

```javascript
// 当前代码：
totalH = 7.2 + 5.1 + 4.5 + 1.2 = 18.3   // ← 注释里写的 18.3 是错的
```

实际执行结果 = `7.2 + 5.1 + 4.5 + 1.2 = 18.0`，与需求 v1.2 一致。只是注释里的等号结果写错了，不影响功能。

**建议**：把注释改成 `= 18.0`，避免后续维护人员困惑。

---

### 2. `computeTotalHeight` 中 support/dorm 分支不会被调用（已知设计）

```javascript
function computeTotalHeight(type, floor, standardHeight = 4.5):
   // ...
   else if type == 'tower': ...
   else if type == 'dorm': ...
   else if floor == 1: ...
   else:  // split / layer / support（N>=2 时）
      return 7.2 + 5.1 + 4.5 * (floor - 2) + 1.2
```

- `support` 和 `dorm` 的高度计算分别在 `buildSupport` 和 `calc_dorm` 中内联完成
- `createConfigs` / 步骤三 / step0 中不会用 `computeTotalHeight` 调用 `support` 或 `dorm`
- 所以 `type == 'dorm'` 分支以及 `support` 走 `else` 分支的情况**当前不会触发**

这是设计上的冗余，不影响功能。但如果未来有人重构时错误地把配套楼/宿舍也走 `computeTotalHeight`，support 2 层会算出 13.5m（正确应为 12.9m）。

**建议**：代码实现时无需修改，但可在 `computeTotalHeight` 上加个注释说明此函数只用于厂房类型（light-steel / split / layer）。

---

### 3. step0 中分层降到 1 层时高度公式不一致（边缘场景）

step0 中分层降级循环：
```javascript
while totalH > Hl and currentFloor > 1:
   currentFloor--
   totalH = 7.2 + 5.1 + 4.5 * (currentFloor - 2) + 1.2
```

当 `currentFloor = 1`：`totalH = 7.2 + 5.1 + 4.5*(-1) + 1.2 = 9.0`

但 `computeTotalHeight('layer', 1)` 或 `computeTotalHeight('split', 1)` 返回 `7.2 + 1.2 = 8.4`。

如果限高 Hl = 8.5m：
- step0 中算得 9.0 > 8.5 → 报错"1 层仍超限"
- 但实际 1 层建筑高度 = 8.4m ≤ 8.5m，**不应报错**

**影响范围**：限高 < 9m 且分层降级到 1 层的极端场景。实际项目中几乎不可能出现。

**建议**：低优先级，可在 step0 中降到 1 层时调用 `computeTotalHeight` 替代内联公式。

---

### 4. `solveIntegerEquation` 单配置搜索范围扩大后的性能

```javascript
// 当前：
searchRange = max(50, floor(n * 0.5))
```

当 n 很大时（如 100 栋），搜索范围 = 50，即搜索 [50, 150]，共 101 个值。这没问题。

但当 n = 500 时，搜索范围 = 250，搜索 [250, 750]，共 501 个值。如果每次外层循环都触发 enrich + 重新求解，可能会有性能问题。

**建议**：代码实现时验证一下性能，如果慢可考虑加个上限（如 `min(200, max(50, n*0.5))`）。
