**状态**：✅ **已修复**

第七轮修订版步骤一 `step1_tower` 已实现完整的分级处理：

1. 初始层数：`max(3, ceil(targetRdArea / areaRef))`

## 三、中等问题（P2）回复

### P2-1：FL 预处理 `FL < 1: break` 与需求语义存在歧义

**状态**：✅ **已修复**

**分析**：
`REQUIREMENTS_v1.2.md` 第 5.7 节明确规定：

```
fixCount = max(1, floor(FL))
```

即即使 `FL < 1`（如 0.8），也固定 1 栋。

但第七轮伪代码 3.3 节原有：

```
if FL < 1: break
```

这与需求语义不一致。

**修复内容**：
已删除 `if FL < 1: break` 条件，统一改为：

```
fixCount = max(1, floor(FL))
```

当 `FL < 1` 时，固定 1 栋最大面积段。若该固定配置导致剩余基底/计容为负，由 `adjustFixedConfigs` 从后往前移除 FL 固定配置，直到剩余为正。

---

### P2-2：`enrichConfigs` 中 `targetBaseApprox` 估算使用 `c.count || 1` 不可靠

**状态**：✅ **已修复**

**分析**：
第七轮伪代码 `enrichConfigs` 中原有：

```
estCount = sum(configs.map(c => c.count || 1))
```

`createConfigs` 创建的 config 没有 `count` 字段，因此 `c.count || 1` 恒为 1，`estCount` 实际上等于 `configs.length`，而非真实预估栋数。

**修复内容**：
1. `enrichConfigs` 函数签名增加 `estimatedTotalCount` 参数：
   ```
   function enrichConfigs(configs, landArea, targetBaseForType, estimatedTotalCount)
   ```

2. 内部估算改为：
   ```
   estCount = max(1, estimatedTotalCount)
   targetBaseApprox = targetBaseForType / estCount
   ```

3. 调用处（`step5_outer_loop` 单一层数分支）传入当前总栋数：
   ```
   totalCount = sum(counts)
   enrichedConfigs = enrichConfigs(targetConfigs, landArea, currentRemainingBase, totalCount)
   ```

---

### P2-3：`solveIntegerEquation` fallback 返回全 0 可能违反"产品类型总栋数 ≥ 2"约束

**状态**：✅ **已修复**

**分析**：
第七轮伪代码中 `solveIntegerEquation` 末尾 fallback 原有：

```
if maxN <= 0:
   return { counts: configs.map(() => 0), totalCap: 0, totalBase: 0 }
```

需求文档要求："数量下限：轻钢/分栋/分层 ≥ 2 栋（如勾选）"。返回全 0 违反了此约束。

**修复内容**：
已将 fallback 改为至少返回 2 栋最小面积段配置：

```
if maxN <= 0:
   // 即使密度可能超限，也保证至少 2 栋最小面积段
   result = configs.map(() => 0)
   minIdx = configs 中 sortedConfigs[0] 的索引
   result[minIdx] = 2
   return {
      counts: result,
      totalCap: sortedConfigs[0].unitCap * 2,
      totalBase: sortedConfigs[0].base * 2
   }
```

返回后由主流程或 `adjustFixedConfigs` 处理可能超限的情况。若仍无法收敛，则向用户提示"参数紧张，密度可能超限"。
