## 一、严重问题（必须修复）

### 1. `step4_core_algorithm` 未处理只有一方配置非空的情况

**问题描述**：
```text
effBh = bhConfigs 的平均效率
effBl = blConfigs 的平均效率
```
如果 `bhConfigs` 为空但 `blConfigs` 非空（或相反），计算 `effBh` 时会除零报错。

**修复建议**：
在函数开头增加分支：
```text
if bhConfigs 为空 且 blConfigs 非空:
   return solveSingleType(blConfigs, remainingCap, remainingBase)
if blConfigs 为空 且 bhConfigs 非空:
   return solveSingleType(bhConfigs, remainingCap, remainingBase)
```

---

### 2. 步骤三 `createConfigs` 未处理 `allTypes.length > 2` 的情况

**问题描述**：
如果 FL 预处理因 `FL < 1` 提前 break，而 `totalLayerTypes > 2`，那么 `allTypes` 可能有 3 种及以上类型。当前代码只处理了 0/1/2 的情况，>2 时 `bhConfigs` 和 `blConfigs` 都为空，导致后续算法错误。

**修复建议**：
- 方案A：FL 预处理 `break` 后，强制只保留效率最高和最低的两种类型作为 Bh/Bl。
- 方案B：即使 `FL < 1` 也继续预处理直到 `totalLayerTypes <= 2`。

---

### 3. `solveIntegerEquation` 的 `baseDiff` 目标值错误

**问题描述**：
```text
baseDiff = |tb - targetBaseLimit|
```
但 `targetBaseLimit = targetBaseBh * 1.05` 是**上限**，不是目标。算法会趋向于把基底做到上限，而不是真实目标 `targetBaseBh`，导致密度偏高或偏低。

**修复建议**：
```text
realTargetBase = targetBaseLimit / 1.05
baseDiff = |tb - realTargetBase|
```

---

### 4. `mapOptimizedToCounts` 未实现

**问题描述**：
`manualDistribute` 返回的优化结果无法映射回 `counts` 数组，外层循环的优化逻辑实际上无法执行。

**修复建议**：
需要明确实现映射逻辑，包括：
- 清空该类型原有的 counts
- 根据 `optimized` 的 base 匹配现有 configs 或添加新 config
- 重新计算 totals

---

### 5. `step5_outer_loop` 中 `enrichConfigs` 的 `targetConfigs` 选择可能为空

**问题描述**：
```text
targetConfigs = (type == 'layer') ? currentBhConfigs : currentBlConfigs
```
如果 `type == 'split'` 且当前是单一层数 split，`currentBlConfigs` 可能为空，导致 `enrichConfigs([], landArea)` 出错。

**修复建议**：
```text
if type == 'layer':
   targetConfigs = currentBhConfigs.length > 0 ? currentBhConfigs : currentBlConfigs
else:
   targetConfigs = currentBlConfigs.length > 0 ? currentBlConfigs : currentBhConfigs
```

## 二、中等问题（建议修复）

### 6. `solveIntegerEquation` DFS 剪枝条件过于激进

**问题描述**：
```text
if hasBaseLimit 且 currentBase > targetBaseLimit * 1.2: return
```
搜索过程中暂时超过 1.2 倍上限可能是正常的，后续配置可能减少。直接 return 可能剪掉最优解。

**修复建议**：
改为 `continue` 或放宽到 1.5 倍。

---

### 7. `step5_outer_loop` 中的 `remainingCap` / `remainingBase` 未更新

**问题描述**：
优化后仍然使用初始的 `remainingCap` 和 `remainingBase` 调用 `solveIntegerEquation`。虽然单一层数时目标值仍是厂房剩余值，但如果已经经过 FL 预处理或固定配置，实际可用的 remaining 可能已经变化。

**修复建议**：
每次优化前重新计算：
```text
remainingBase = targetBase - fixedBase - usedBase
remainingCap = targetCap - fixedCap - usedCap
```

---

### 8. `fineTune` 无法解决初始密度严重不足的问题

**问题描述**：
如果初始状态密度就远低于目标（之前测试发现的 -25% ~ -50%），`fineTune` 的调节方向是增加栋数，但受 `targetBase * 1.05` 上限限制，无法大幅增加。

**修复建议**：
核心问题在 `solveIntegerEquation` 的评分函数（见问题3）。修复后应该能缓解。

---

### 9. `enrichConfigs` 中轻钢效率计算简化

**问题描述**：
```text
eff = type == 'light-steel' ? 2.1 : floor
```
轻钢实际效率 = `(base * 2 + 200) / base = 2 + 200/base`，不是常数 2.1。对大 base 接近 2.1，对小 base 偏差大。

**修复建议**：
对每个候选面积单独计算实际效率：
```text
if type == 'light-steel':
   eff = (area * 2 + 200) / area
else:
   eff = floor
```

## 三、小问题（边界或文档）

### 10. `buildSupport` 中极端小面积的偏差计算

**问题描述**：
如果 `supportCap` 很小（如 100），`finalSupportCap` 取 10 倍数后仍可能偏差 > 0.2%。

**修复建议**：
明确规则：如果三次取整后仍不满足，是否继续取 5 倍数或直接输出。

---

### 11. 输出合并时缺少 `totalCap` 字段

**问题描述**：
主流程输出：
```text
output.push({
   ...config,
   count: count,
   unitCap: config.unitCap,
   ...
})
```
没有 `totalCap` 字段，前端需要额外计算。

**修复建议**：
输出时增加 `totalCap = config.unitCap * count`。

---

### 12. `solveIntegerEquation` 单配置搜索范围可能过大

**问题描述**：
单配置时搜索范围 `n - 200` 到 `n + 200`，共 401 个值。对于小目标值可能过大，但对于大目标值可能不够。

**修复建议**：
根据目标栋数动态调整范围，或保持当前范围（性能影响小）。

## 四、关键问题根因分析

之前 11 组测试出现密度严重不足（差距 25% ~ 50%）的核心原因：

1. **`maxLayerHeight` 是层数而非高度**：已修复。
2. **`solveIntegerEquation` 评分函数只优化计容，基底权重太低且目标值错误**：这是当前最严重的问题，对应本报告问题3。
3. **双拼转换导致 `actualBase` 与 `base` 不一致**：已通过"全部按独栋处理"规则解决。

修复本报告中问题1-5后，预计密度问题能大幅改善。
