## 配套用房规则（核心！）

### 规则1：配套用房总面积 ≤ 2400
- 全部给配套楼
- 配套楼基底 = 总面积/3，向下取50倍数，偏差>0.2%则取20倍数，仍>0.2%则取10倍数
- 不配置配套宿舍

### 规则2：配套用房总面积 > 2400
- 必须同时配置配套楼和配套宿舍（或仅配套宿舍，取决于 UI 选择）
- 配套楼面积强制限定在 **1200-2400** 区间内
  - 总面积 < 3600 → 配套楼面积 = 1200
  - 总面积 >= 3600 → 配套楼面积 = 用户选择的配套楼面积段，钳制在 [1200, 2400]
- 配套宿舍面积 = 总面积 - 配套楼面积
- 配套宿舍强制层数 ≥ 2层

### 规则3：配套用房总面积 < 1200（极端情况）
- 全部分配给配套楼
- 配套楼层数改为 **2层**（不是3层）
- 建筑高度 = 7.2 + 4.5 + 1.2 = 12.9m
- 配套楼基底 = 总面积/2，向下取50倍数，偏差>0.2%则取20倍数，仍>0.2%则取10倍数

### 规则4：配套楼面积边界总结
| 总面积 | 配套楼面积 | 配套宿舍 | 配套楼层数 |
|--------|-----------|---------|-----------|
| < 1200 | 全部（可<1200） | 无 | 2层 |
| 1200-2400 | 全部 | 无 | 3层 |
| 2400-3600 | 1200（固定） | 剩余 | 3层 |
| 3600+ | 2400（上限） | 剩余 | 3层 |

### 规则5：UI提醒
- 当R2>0时，用户必须选择配套楼或配套宿舍，否则提示"配套用房未选择"
- 当配套面积>2400且用户只选了配套楼（没选配套宿舍），提示"配套用房面积未用尽，请勾选配套宿舍"

## 分层厂房限高处理规则（新增！）
- 当分层厂房高度超过限高时，**自动逐层减1**
- 允许降到任意层数（无6层下限）
- ≥5层：归类为分层，备注"因限高原因，分层降低层数"
- ≤4层：归类为分栋，面积段与原分栋并集（取消双拼，还原原始值）
- 此规则适用于所有分层厂房配置（FL预处理、Bh/Bl配置、createBhBlConfigs）

## 关键规则
1. **3.5层处理**：3.5 vs 3 → 3.5是Bh；3.5 vs 4 → 3.5是Bl
2. **轻钢等效层数**：2.1（计容计算用）
3. **轻钢计容**：独栋=基底×2+200，双拼=总基底×2+400
4. **分层双拼**：~~600/800强制双拼，1000/1200不强制；丰富面积段后≤800也强制双拼~~ **全部按独栋处理**
5. **FL预处理**：分层厂房公式 /2，可循环执行直到层数类型≤2
6. **三种厂房类型**：强制分配1栋最低层数最小面积分层，剩余退化为分栋+轻钢
7. **丰富面积段触发**：遍历算法后最优解仍无法满足容积率允许误差时自动触发
8. **双拼转换规则**：算法推导全程按独栋计算（分层≤800强制双拼除外），最终输出时才做双拼转换
   - 偶数栋 → 全部转为双拼（栋数÷2）
   - 奇数栋 → floor(栋数÷2)双拼 + 1独栋
   - 分层≤800强制双拼的类型跳过此转换

## 丰富面积段规则
- 触发条件：遍历算法后最优解仍无法满足容积率允许误差
- 增加上限：≤20000m²→+1，≤50000m²→+2，>50000m²→+3
- 方法：递减（x-50, x+50, x-100）+ 内插（相邻两值的中间值）
- 取整：所有新增值必须是50的倍数
- 去重：新增值不能与已有值重复

## 双拼规则（2026-07-08 最终简化版）

**核心原则：算法推导全程全部按独栋处理，不考虑双拼转换。**

1. **前置处理**：不转换600/800，保持原始面积段（600/800/1000/1200）
2. **算法推导**：所有产品（轻钢、分栋、分层）统一按独栋逻辑
   - `base = area`（原始面积段）
   - `unitCap = base * floors`（分层/分栋）或 `base * 2 + 200`（轻钢）
   - `solveIntegerEquation` 中 `testBase = c.base * nn`（原始基底×栋数）
3. **丰富面积段**：基于原始面积段（如600→550/650，800→750/850）
4. **最终输出**：暂不处理双拼转换，直接输出独栋配置
   - 等算法跑通、11组场景测试通过后，再单独处理"如何把独栋输出改为双拼"
5. **降级处理**：分层降级≤4层归类为分栋时，面积段直接合并（无需还原双拼）

**删除的内容**：
- `duplexSourceMap` 全局变量及数据流
- `preprocessLayerAreas` 中的面积段转换逻辑
- `createConfigs` 中的 `duplexSource` 字段
- `convertToDuplex` 中的双拼转换逻辑（保留函数框架，内部简化为直接输出）
- `getActualBase` 函数（不再需要，直接用 `c.base * count`）

## 2026-07-09 产业大厦限高超限处理规则更新

### 新增规则
产业大厦超限高时，采用"先降标准层层高、再减层数"的分级处理逻辑，避免直接减层导致面积段过大、占地过多。

### 详细流程
1. **确定初始层数**：`floors = max(3, ceil(targetRdArea / areaRef))`
2. **默认 4.5m 标准层计算高度**：`totalH = 6.6 + 4.5 * (floors - 1) + 1.2`
3. **超限后先降标准层到 4.2m**：`totalH = 6.6 + 4.2 * (floors - 1) + 1.2`
   - 若此时不超限 → 采用此方案，层数不变
4. **仍超限则恢复 4.5m 并减层数**：`floors = max(3, floor((Hl - 6.6 - 1.2) / 4.5) + 1)`
5. **安全兜底**：若减到 3 层仍超限 → 报错"限高不足，无法放置3层产业大厦"

### 数据结构更新
产业大厦配置输出增加字段：
- `standardHeight`: 记录最终确定的标准层层高，值为 4.5 或 4.2，供前端产品配置表展示

## 2026-07-09 manualDistribute 改进方案（已确认）

### 用户要求

1. **取整精度优化**：中间面积段 `c` 的取值精度从固定 10 倍优化为分级降级（10 → 5 → 2），并在已经是目标倍数时跳过取整。
2. **分支细化**：增加 4 个分支，覆盖更大栋数范围，通过增大 `bb` 来控制分布不均。

### 新分支结构

| 分支 | 条件 | aa | bb | cc |
|------|------|----|----|----|
| 1 | `S < b` | 1栋，base=ceil(S/10)*10 | - | - |
| 2 | `refCount <= 3` | 统一 base=ceil(S/N/10)*10 | - | - |
| 3 | `3 < N <= 6` | `ceil(2N/3) - 2` | 1 | `N - aa - bb` |
| 4 | `6 < N <= 11` | `ceil(2N/3) - 2` | 2 | `N - aa - bb` |
| 5 | `11 < N <= 20` | `ceil(2N/3) - 3` | 3 | `N - aa - bb` |
| 6 | `20 < N <= 32` | `ceil(2N/3) - 4` | 4 | `N - aa - bb` |
| 7 | `32 < N <= 47` | `ceil(2N/3) - 5` | 5 | `N - aa - bb` |
| 8 | `N > 47` | `ceil(2N/3) - 6` | 6 | `N - aa - bb` |

### 防分布不均证明

触发条件：`aa > 2*(bb+cc)` 或 `bb > 2*(aa+cc)`

由于 `cc = N - aa - bb`，等价于 `aa > 2N/3` 或 `bb > 2N/3`。

新规则中 `aa = ceil(2N/3) - k`，其中 k ≥ 2：
- `aa <= 2N/3 + 1 - k <= 2N/3 - 1 < 2N/3`
- `bb = k <= 6`，对于 N > 6 的场景，远小于 2N/3

**结论**：新规则下 manualDistribute 输出天然不触发分布不均。

### c 的取整规则

```
remaining = S - a*aa - b*bb
c_raw = remaining / cc
```

1. **10 倍精度**：若 c_raw 不是 10 的倍数，c = ceil(c_raw/10)*10；否则保持
2. **边界校验**：用全局 totals 检查容积率/密度是否超标
3. **5 倍精度**：若超标且 c_raw 不是 5 的倍数，c = ceil(c_raw/5)*5；否则保持
4. **2 倍精度**：若仍超标且 c_raw 不是 2 的倍数，c = ceil(c_raw/2)*2；否则保持

### 实现分工

- `manualDistribute`：生成按规则分布的配置（面积段 + 栋数），支持按精度参数生成候选
- `step5_outer_loop` / `step5_single_layer_loop`：负责全局边界校验和精度降级回调

### 关联优化

1. 同步增强 `fineTuneIgnoreDensity`（单一层数）和 `fineTune`（Bh/Bl）的轮数和组合搜索。
2. 同步扩大 `solveIntegerEquation` 搜索范围，单一层数时放宽 cap 剪枝。
3. 强制分布不均检查：优化后必须再次 `checkDistribution`，触发则不接受。

## 2026-07-09 晚间 - 单一层数独立算法讨论与实现中

###  tonight 讨论结论

用户明确：
1. **单一层数定义**：去掉固定产品（产业大厦、配套楼、配套宿舍）后，厂房部分只剩 **一种层数类型**（同一种产品且同一层数）。此时应走独立算法，不走 Bh/Bl。
2. **特例**：即使厂房有两种层数，建立 Bh/Bl 后，若 `avgEff = remainingCap / remainingBase` 超出 `[BlEff, BhEff]` 范围，则取消偏离平均效率更多的那种层数，退化回单一层数流程。
   - `avgEff >= BhEff`：取消 Bl，只保留 Bh
   - `avgEff <= BlEff`：取消 Bh，只保留 Bl
   - 该特例既适用于同产品的两个层数，也适用于不同产品组合，作为兜底保证算法自洽。
3. **单一层数验收**：
   - 完全不看建筑密度
   - 优先满足容积率
   - 仅检查：容积率是否在允许范围内、面积段分布是否不均

### 实现方案

1. 新增 `step4_single_layer(configs, remainingCap, remainingBase)`：
   - 调用 `solveIntegerEquation(remainingCap, configs, Infinity)` 只优化容积率
   - 修复 `solveIntegerEquation` 对 `Infinity` 的处理：当 `targetBaseLimit === Infinity` 时视为无密度限制，避免 `maxN = Infinity` 导致死循环

2. 新增 `step5_single_layer_loop(...)`：
   - 只看容积率和分布
   - 不看建筑密度
   - 专用 `fineTuneIgnoreDensity` 只调节容积率

3. 主流程中增加判断：
   - `uniqueLayerTypes.size === 1`：自然单一层数
   - `uniqueLayerTypes.size === 2`：检查是否退化
   - 否则走 Bh/Bl

## 2026-07-10 FL 预处理规则修正（示例2问题）

### 问题

示例 2（轻钢 + 分栋 2 层 + 分栋 3.5 层）中，用户选择了轻钢厂房，但算法输出完全没有轻钢。原因是：

1. 当前代码把 FL 预处理的目标错误地限定为 `splitFloors + layerFloors ≤ 2`，没有考虑轻钢带来的额外效率类型。
2. 当前 FL 预处理公式 `fixCount = max(1, floor(FL))` 在 FL 较大时会固定过多栋数，挤压了后续 Bh/Bl 的调节空间。
3. 当 `allTypes.length > 2` 时，当前代码直接取效率最高和最低的类型作为 Bh/Bl，中间类型被丢弃，导致轻钢被直接淘汰。

### 修正后的 FL 预处理规则

#### 触发条件

FL 预处理应在「所有厂房效率类型数量 > 2」时触发。效率类型包括：
- 轻钢厂房（效率 2.1）
- 分栋厂房每一种层数
- 分层厂房每一种层数

#### 处理顺序

按效率从高到低排序，**每次固定效率最高的类型**，直到剩余效率类型数量 ≤ 2。

#### 固定规则（分栋厂房）

针对当前待处理的最高层数分栋：

```
a = 该层数最高面积段 × 层数
FL = remainingCap / a
```

| FL 范围 | 固定策略 |
|---|---|
| FL ≤ 4 | 该层数最高面积段固定 1 栋 |
| 4 < FL ≤ 6 | 该层数各面积段各固定 1 栋 |
| 6 < FL ≤ 8 | 该层数各面积段按面积由高到低分配 1, 1, …, 2 栋 |
| FL > 8 | 该层数各面积段各固定 2 栋 |

固定完成后，**该层数类型整体移除**，不再参与后续 Bh/Bl。

#### 固定规则（分层厂房）

与分栋规则逻辑相同，但 FL 计算需除以 2：

```
a = 该层数最高面积段 × 层数
FL = remainingCap / a / 2
```

| FL 范围 | 固定策略 |
|---|---|
| FL ≤ 4 | 该层数最高面积段固定 1 栋 |
| 4 < FL ≤ 6 | 该层数各面积段各固定 1 栋 |
| 6 < FL ≤ 8 | 该层数各面积段按面积由高到低分配 1, 1, …, 2 栋 |
| FL > 8 | 该层数各面积段各固定 2 栋 |

同样，固定完成后该层数类型整体移除。

#### 轻钢厂房

轻钢厂房效率通常为 2.1，在常见组合中属于较低效率类型，**不单独设置 FL 预处理公式**。它通常作为剩余两种类型之一进入 Bh/Bl。

#### 三种厂房类型处理保留

当用户同时选择 `轻钢 + 分栋 + 分层` 时，仍保留原 3.4 节逻辑：
- 强制分配 1 栋最低层数最小面积的分层
- 该分层即为全部分层，剩余退化为分栋 + 轻钢
- 若分栋仍有多种层数，再固定最高层数的最大面积段 1 栋

处理完后，剩余 ≤ 2 种效率类型进入 Bh/Bl。

#### 修正后的步骤三流程

```
1. 构建 allTypes = [light-steel(2.1)] + split各层数 + layer各层数
2. 当 allTypes.length > 2 时：
   a. 按效率从高到低排序
   b. 取最高效率的类型
   c. 如果是 split/layer 且有多个层数：
      - 用对应的 FL 规则计算固定栋数
      - 生成固定配置，计入 fixedConfigs
      - 从 allTypes 中移除该层数
   d. 如果是轻钢：保留，不固定（因为轻钢效率偏低，通常不会成为最高效率）
   e. 重复直到 allTypes.length ≤ 2
3. 剩余 2 种类型进入 Bh/Bl（1 种类型走单一层数流程）
4. 删除原代码中「allTypes.length > 2 时取头尾丢中间」的分支
```

### 示例 2 按修正规则估算

输入：
- S=50000, F=1.2, D=45%, R2=15%
- 固定产品：配套楼 1800 + 配套宿舍 7200 → 占用基底 1500，计容 9000
- 厂房剩余：基底 21000，计容 51000
- 效率类型：split-3.5(3.5)、light-steel(2.1)、split-2(2.0)

FL 预处理：
- 最高效率 = split-3.5
- a = 670 × 3.5 = 2345
- FL = 51000 / 2345 ≈ 21.75 > 8
- 固定 2 栋 670m² split-3.5 + 2 栋 400m² split-3.5
- 占用基底 2140，计容 7490

剩余：
- remainingBase = 21000 - 2140 = 18860
- remainingCap = 51000 - 7490 = 43510
- avgEff = 43510 / 18860 ≈ 2.307

剩余 2 种进入 Bh/Bl：
- Bh = light-steel（效率约 2.1+）
- Bl = split-2（效率 2.0）

## 2026-07-10 FL 预处理触发条件回归确认

### 讨论结论

由于第八轮算法中已将「分栋 + 轻钢」组合作为独立分支（方案 B）处理，所有含轻钢的有效组合都不会进入通用 FL 预处理。因此通用 FL 预处理的触发条件可以回归到原始设计：

> **通用 FL 预处理仅统计 `splitFloors.length + layerFloors.length`，当 > 2 时触发。**

轻钢厂房不纳入通用 FL 预处理的统计范围。

### 修正后的步骤三流程

```
步骤三：厂房预处理
├─ 分栋 + 轻钢组合 → 方案 B
│   ├─ 有分层时：先固定 1 栋最低层数最小面积分层
│   ├─ 对较低层数分栋做手动分配（4/6/8 阈值）
│   ├─ B1：只剩轻钢 → 单一层数
│   └─ B2：轻钢 + 较高层数分栋 → 参考方案 A 逻辑
│
├─ 四层分栋 + 六层分层（无轻钢）→ 方案 A
│
└─ 通用 FL 预处理（仅 split/layer 层数类型 > 2 时触发）
    ├─ 优先分层，其次分栋
    ├─ 每次固定当前最高层数类型
    ├─ 按 4/6/8 阈值分配栋数
    └─ 直到 splitFloors.length + layerFloors.length ≤ 2
```

### 删除「取头尾丢中间」分支

第八轮伪代码中彻底删除以下逻辑：

```js
// 第七轮的错误逻辑
else:
  // allTypes.length > 2：取效率最高的作为 Bh，效率最低的作为 Bl
  allTypes.sort((a, b) => 效率(b) - 效率(a))
  bhType = allTypes[0]
  blType = allTypes[allTypes.length - 1]
  // 中间类型被丢弃
```

原因：该分支会粗暴地丢弃中间效率类型，导致用户选择的产品（如轻钢）在输出中消失。

### 通用 FL 预处理后 allTypes 的状态

由于轻钢不进入通用 FL 预处理，且 FL 预处理会循环直到 split/layer 层数类型 ≤ 2，因此：

```js
allTypes.length 只可能为 0、1 或 2
```

- 0：无厂房
- 1：单一层数
- 2：正常 Bh/Bl

不再出现 `allTypes.length > 2` 的情况。


## 2026-07-10 B2 分支动态 Bh/Bl 修正

### 问题
在「分栋 + 轻钢」组合的 B2 分支中，原代码固定把分栋设为 Bh、轻钢设为 Bl。当分栋层数为 2（效率约 2.0）而轻钢效率（约 2.1）更高时，Bh/Bl 关系被颠倒，导致 step4 密度调整后错误地抛弃了轻钢。

### 修复
B2 分支现在按实际平均效率动态分配 Bh/Bl：

```js
splitEff = splitConfigs.length
  ? splitConfigs.reduce((s, c) => s + c.unitCap / c.base, 0) / splitConfigs.length
  : higherFloor
lightSteelEff = lightSteelConfigs.length
  ? lightSteelConfigs.reduce((s, c) => s + c.unitCap / c.base, 0) / lightSteelConfigs.length
  : 2.1

if (lightSteelEff > splitEff) {
  bhConfigs = lightSteelConfigs
  blConfigs = splitConfigs
} else {
  bhConfigs = splitConfigs
  blConfigs = lightSteelConfigs
}
```
