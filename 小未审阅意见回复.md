## 逐条回复

### 意见 5：`enrichConfigs` 用全局基底估算类型基底

**状态**：✅ 已修复

**问题确认**：存在。`totalTargetBase = landArea * D` 是整个项目的目标基底，而 `enrichConfigs` 只用于某一类型，用全局目标基底估算不合理。

**修复方式**：
- 函数签名改为 `(configs, landArea, targetBaseForType)`
- `targetBaseApprox = targetBaseForType / max(1, estCount)`
- 调用处传入 `currentRemainingBase`（单类型 enrich 时即为该类型的实际目标基底）

### 意见 6：`calc_dorm` 钳制后未二次调整层数

**状态**：✅ 已修复

**问题确认**：存在。`finalBase` 被钳制到 [600, 1200] 后，`unitArea = finalBase * floors` 可能偏离 `dormCap`。

**修复方式**：
在钳制后增加二次调整：
```
floors = max(2, ceil(dormCap / finalBase))
totalH = 4.8 + 3.6 * (floors - 1) + 1.2
if totalH > referenceHeight:
   totalH = 4.8 + 3.3 * (floors - 1) + 1.2
   if totalH > referenceHeight:
      floors = max(2, floor((referenceHeight - 4.8 - 1.2) / 3.3) + 1)
      totalH = 4.8 + 3.3 * (floors - 1) + 1.2
```

### 意见 7：输入校验层数检查在 `step0` 之前

**状态**：⚠️ 确认存在，但定位为低风险 / UI 兜底

**问题确认**：理论上存在。如果分层降级后合并到分栋，导致分栋层数超过 2 种，输入校验会漏检。

**处理说明**：
- UI 已限制用户分栋/分层最多各选 2 种层数
- 算法中 `allTypes.length > 2` 有兜底处理（取效率最高/最低两种）
- 当前不修改，因为真实场景极少触发，且已有兜底

---

### 意见 8：`step0` 分层 1 层降级到分栋的公式不兼容

**状态**：✅ 已修复

**问题确认**：存在。`computeTotalHeight` 对 `floor == 1` 使用 `7.2 + 5.1 + 4.5*(1-2) + 1.2 = 9.0m`，不符合单层逻辑。

**修复方式**：
在 `computeTotalHeight` 中增加 `floor == 1` 分支：
```
else if floor == 1:
   return 7.2 + 1.2  // 单层分栋/分层
```

---

### 意见 9：`solveIntegerEquation` k=1 时搜索范围过大

**状态**：✅ 已优化

**问题确认**：存在。`n - 200` 到 `n + 200` 范围过大，小项目时很多计算被跳过。

**修复方式**：
改为动态范围：
```
searchRange = max(50, floor(n * 0.5))
startN = max(1, n - searchRange)
endN = n + searchRange
```

### 意见 10：`fineTune` 与 `step5` 循环叠加

**状态**：✅ 已优化

**问题确认**：存在。`step5` 10 轮 × `fineTune` 10 轮 = 最多 100 轮，极端场景可能偏慢。

**修复方式**：
`MAX_FINE_TUNE` 从 10 降为 5。

### 意见 11：`adjustFixedConfigs` 兜底策略激进

**状态**：✅ 已决策，维持当前策略

**问题确认**：存在。移除所有 FL 固定配置后仍为负就报错，可能过于激进。

**用户决策**：**不允许减少产业大厦面积**。

**处理说明**：
- 维持当前策略：仅允许移除 FL 预处理中的固定配置
- 移除所有 FL 固定配置后仍超密度/容积率上限 → 直接报错
- 产业大厦面积不得作为兜底调整项
