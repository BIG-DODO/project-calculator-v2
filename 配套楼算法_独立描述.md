# 配套楼配置算法（独立描述）

> **版本**：2026-07-09 简化版
> **核心规则**：
> - `totalAncillary ≤ 2400`：用户可任意选择（仅配套楼 / 仅配套宿舍 / 两者都选）
> - `totalAncillary > 2400`：强制必须选配套宿舍，配套楼可选
> - 基底取整：50 倍数 → 20 倍数 → 10 倍数（每次检查偏差是否 > 0.2%）

---

## 一、输入

| 参数 | 含义 |
|------|------|
| `totalAncillary` | 配套用房目标计容面积 = `S * F * R2` |
| `selectedProducts` | 用户选择的产品类型 Set |
| `supportAreaRef` | 用户选择的配套楼面积段（参考值，不作为定值） |

---

## 二、输出

返回一个对象：

```javascript
{
   config: {
      id: 'support',
      type: '配套楼',
      base,           // 配套楼基底
      unitCap,        // 配套楼计容面积
      floors,         // 层数（2 或 3）
      totalHeight,    // 建筑高度
      count: 1
   },
   supportCap,        // 配套楼最终分配的计容面积
   supportBase        // 配套楼基底
}
```

若用户未选择配套楼，返回 `null`。

## 三、主流程

```text
function calculateSupport(totalAncillary, selectedProducts, supportAreaRef):

   // 用户未选择配套楼
   if selectedProducts 不包含 'support':
      return null

   // 情况1：totalAncillary ≤ 2400，用户仅选择配套楼
   if totalAncillary <= 2400 且 selectedProducts 不包含 'dorm':
      return buildSupport(totalAncillary)

   // 情况2：totalAncillary ≤ 2400，用户同时选择配套楼和配套宿舍
   // UI逻辑：两者都选时默认合建为一栋配套楼，全部分配给配套楼
   if totalAncillary <= 2400 且 selectedProducts 包含 'dorm':
      return buildSupport(totalAncillary)

   // 情况3：totalAncillary > 2400，用户仅选择配套宿舍（未选配套楼）
   // 理论上不会进入本函数，因为UI已阻止此组合
   if totalAncillary > 2400 且 selectedProducts 不包含 'dorm':
      return null

   // 情况4：totalAncillary > 2400，用户同时选择配套楼和配套宿舍
   if totalAncillary > 2400 且 selectedProducts 包含 'dorm':
      // 配套楼面积强制限定在 [1200, 2400]
      if totalAncillary < 3600:
         supportCap = 1200
      else:
         supportCap = 2400
      return buildSupport(supportCap)
```

## 四、子函数：buildSupport

```text
function buildSupport(supportCap):

   if supportCap < 1200:
      // 极端情况：总面积 < 1200
      supportFloors = 2
      totalH = 7.2 + 4.5 + 1.2 = 12.9
      exactBase = supportCap / 2
   else:
      // 正常情况
      supportFloors = 3
      totalH = 7.2 + 5.1 + 4.5 + 1.2 = 18.3
      exactBase = supportCap / 3

   // 第一次取整：50 的倍数
   supportBase = floor(exactBase / 50) * 50
   finalSupportCap = supportBase * supportFloors

   // 第一次偏差检查
   if |finalSupportCap - supportCap| / supportCap > 0.002:
      // 第二次取整：20 的倍数
      supportBase = floor(exactBase / 20) * 20
      finalSupportCap = supportBase * supportFloors

      // 第二次偏差检查
      if |finalSupportCap - supportCap| / supportCap > 0.002:
         // 第三次取整：10 的倍数
         supportBase = floor(exactBase / 10) * 10
         finalSupportCap = supportBase * supportFloors

   return {
      config: {
         id: 'support',
         type: '配套楼',
         base: supportBase,
         unitCap: finalSupportCap,
         floors: supportFloors,
         totalHeight: totalH,
         count: 1
      },
      supportCap: finalSupportCap,
      supportBase: supportBase
   }
```

## 五、配套宿舍目标面积

配套宿舍只在以下情况配置：

```text
// 情况A：totalAncillary > 2400 且用户选了配套宿舍
if totalAncillary > 2400 且 selectedProducts 包含 'dorm':
   dormCap = totalAncillary - supportCap
   // 调用 calc_dorm(dormCap, referenceHeight)

else if totalAncillary <= 2400 且 selectedProducts 包含 'dorm' 且 不包含 'support':
   dormCap = totalAncillary
   // 调用 calc_dorm(dormCap, referenceHeight)
```

## 六、关键说明

1. **配套楼不是定值**：用户选择的 `supportAreaRef` 仅作为参考，最终配套楼面积由 `totalAncillary` 和 UI 选择组合决定。

2. **`totalAncillary ≤ 2400` 且两者都选时**：全部给配套楼，不分配宿舍。产品逻辑上可理解为"配套楼和宿舍合建为一栋楼"，算法上只输出一栋配套楼。

3. **`totalAncillary > 2400` 且两者都选时**：配套楼固定为 1200 或 2400，剩余面积全给宿舍。

4. **配套楼面积边界**：
   - `< 1200`：2 层
   - `1200 - 2400`：3 层
   - `> 2400`：强制截断为 2400

5. **基底取整规则**：
   - 先取 50 的倍数
   - 偏差 > 0.2% 时取 20 的倍数
   - 偏差仍 > 0.2% 时取 10 的倍数

6. **宿舍强制层数 ≥ 2**：无论 `totalAncillary` 多小，只要配置宿舍，层数至少 2 层。

## 七、配套宿舍算法（calc_dorm）

```text
function calc_dorm(dormCap, referenceHeight):

   estFloors = max(2, floor((referenceHeight - 4.8 - 1.2) / 3.6) + 1)
   singleArea = dormCap / estFloors

   if singleArea < 600:
      base = 600
   else if singleArea > 1200:
      base = 1200
   else:
      base = round(singleArea / 100) * 100

   floors = max(2, ceil(dormCap / base))
   totalH = 4.8 + 3.6 * (floors - 2) + 1.2

   if totalH > referenceHeight:
      totalH = 4.8 + 3.3 * (floors - 2) + 1.2
      if totalH > referenceHeight:
         floors = max(2, floor((referenceHeight - 4.8 - 1.2) / 3.3) + 2)
         totalH = 4.8 + 3.3 * (floors - 2) + 1.2

   exactBase = dormCap / floors
   finalBase = floor(exactBase / 20) * 20
   unitArea = finalBase * floors

   return {
      config: {
         id: 'dorm',
         type: '配套宿舍',
         base: finalBase,
         unitCap: unitArea,
         floors,
         totalHeight: totalH,
         count: 1
      },
      base: finalBase,
      cap: unitArea
   }
```

## 八、边界条件检查清单

1. `totalAncillary = 0` 时，是否跳过配套用房计算？
2. `totalAncillary < 1200` 时，配套楼 2 层的基底取整是否正确？
3. `totalAncillary = 1200` 时，是否按 3 层处理？
4. `totalAncillary = 2400` 时，是否仍按"≤2400"分支处理？
5. `totalAncillary = 3600` 时，配套楼是否固定为 1200 还是 2400？（根据规则：`< 3600` 为 1200，`>= 3600` 为 2400）
6. 三次取整后偏差是否仍可能 > 0.2%？若出现，是否需要兜底处理？
7. 宿舍在 `totalAncillary <= 2400` 且仅选宿舍时，是否按完整 `totalAncillary` 计算？
