#### 问题1：振荡检测中distCheck字段访问风险（运行时错误）

**位置**: ALGORITHM(2).md 子步骤5.3 第876-880行

**问题描述**:
```javascript
if distCheck.needsOptimization or detectOscillation():
   type = distCheck.type      // ← 可能为undefined！
   a = distCheck.a            // ← 可能为undefined！
   b = distCheck.b            // ← 可能为undefined！
   eff = distCheck.eff        // ← 可能为undefined！
```

`checkDistribution()`在不需要优化时返回 `{needsOptimization: false}`，此时**没有type/a/b/eff字段**。

当`detectOscillation()=true`但`distCheck.needsOptimization=false`时，条件成立进入if分支，但后续访问`distCheck.type`等字段会得到**undefined**，导致`manualDistribute(undefined, undefined, undefined, undefined)`运行错误。

**触发场景**: 分布均衡但密度在±5%边界振荡时（恰好需要优化的面积段分布合理，但密度来回跳动）。

**修复建议**:
```javascript
// 方案A：分离处理
if distCheck.needsOptimization:
   type = distCheck.type
   a = distCheck.a
   b = distCheck.b
   eff = distCheck.eff
   // 正常执行优化...
elif detectOscillation():
   // 振荡但分布均衡：需要手动收集type信息
   // 从allConfigs中找到主要类型
   type = findDominantType(finalCounts, allConfigs)
   // 重新计算a/b/eff...
```

```javascript
// 方案B（更简洁）：确保checkDistribution在needsOptimization=false时也返回基本字段
checkDistribution返回值统一为：
{
   needsOptimization: true/false,
   type: dominantType,   // 始终返回主要类型（即使不需要优化）
   a: minBase,
   b: maxBase,
   eff: avgEfficiency
}
```

#### 问题2：子步骤0.0数据流未闭环 — processedAreas和duplexSourceMap未明确传递到后续步骤

**位置**: ALGORITHM(2).md 子步骤0.0 输出 → 子步骤0.5/3.5/3.6 输入

**问题描述**:

0.0节preprocessLayerAreas输出两个关键数据：
- `processedAreas`：转换后的面积段数组（如[1000,1200,1200,1600]）
- `duplexSourceMap`：转换映射关系（{1200:600, 1600:800}）

但文档中**没有明确说明**这两个数据如何传递到后续步骤：

1. **processedAreas是否更新了productOptions['layer'].areas？**
   - 0.5节说"更新productOptions"，但只提到floors，未提areas
   - 如果未更新，后续createConfigs（3.5节）从`productOptions['layer'].areas`读取的仍是原始值[600,800,1000,1200]
   - 如果已更新，需要明确写出更新逻辑

2. **duplexSourceMap如何在createConfigs中设置duplexSource？**
   - 3.5节createConfigs中设置了`duplexSource: (type == 'layer' && area <= 800) ? area : null`
   - 但如果areas已经是转换后的值（如1200），area=1200>800，duplexSource=null
   - 转换来源信息丢失！

3. **duplexSourceMap如何传递到convertToDuplex？**
   - 3.6节签名：`convertToDuplex(counts, configs, duplexSourceMap)`
   - 但文档中调用convertToDuplex的位置（步骤五末尾）未说明duplexSourceMap的来源

**核心矛盾**: 
- 如果0.0转换了areas但不更新productOptions，后续步骤看到的仍是原始值，0.0白做
- 如果0.0转换了areas并更新productOptions，createConfigs中area是转换后的值（1200），duplexSource判断（area <= 800）永远不会触发，duplexSource永远为null
- 无论哪种情况，3.6节convertToDuplex中`c.duplexSource`要么为空（未传递），要么为null（条件不满足），**源面积映射信息丢失**

**修复建议**: 明确数据流

```javascript
// === 子步骤0.0 输出 ===
return {
   areas: processedAreas,           // 转换后的面积段（排序不去重）
   duplexSourceMap: duplexSourceMap  // 映射关系
}

// === 子步骤0.5 更新 ===
productOptions['layer'].areas = preprocessResult.areas
duplexSourceMap = preprocessResult.duplexSourceMap  // 保存为全局变量

// === 子步骤3.5 createConfigs修改 ===
function createConfigs(type, floor):
   configs = []
   areas = productOptions[type].areas
   
   for each area in areas:
      if type == 'layer':
         // 从duplexSourceMap判断是否是转换来的面积段
         sourceArea = duplexSourceMap[area] || null
         if sourceArea != null:
            base = area  // 已经是双拼后的基底（如1200）
            duplexSource = sourceArea  // 600
         else:
            base = area  // 原生面积段（如1000/1200）
            duplexSource = null
      else:
         base = area
         duplexSource = null
      
      // ... 后续逻辑不变
      configs.push({
         id: type, type: typeName, base, unitCap,
         originalArea: area,        // 算法中使用的面积（可能是转换后的）
         duplexSource: duplexSource, // 强制双拼的原始面积（600/800），原生为null
         floors: floor, totalHeight: totalH
      })

// === 子步骤3.6 convertToDuplex ===
function convertToDuplex(counts, configs):
   // duplexSourceMap从全局变量读取（或在调用时传入）
   for i, c in configs:
      if c.id == 'layer' and c.duplexSource != null:
         // 来自600/800转换：显示duplexSource（600/800）
         displayBase = c.duplexSource
      else if c.id == 'layer':
         // 原生：显示base（1000/1200）
         displayBase = c.base
      // ...
```

#### 问题3：enrichConfigs中area与base的映射关系不明确

**位置**: ALGORITHM(2).md 子步骤5.5.2/5.5.3

**问题描述**:

5.5.2节"生成新增面积段"说"对configs中每个现有配置"，但configs中的面积信息是`base`字段，不是原始`area`。对于分层厂房：
- area=600时base=1200
- area=800时base=1600
- area=1000时base=1000
- area=1200时base=1200

递减法需要从原始area生成候选（如从600生成550），但config中只有base（1200），需要反推area=base/2=600。这增加了复杂度且容易出错。

5.5.3节的效率计算：
```javascript
eff = area * floor / (type == 'layer' && area <= 800 ? area * 2 : area)
```

这里的`area`如果是base（1200），判断`1200 <= 800`为false，分母=1200，效率=floor。
但如果原始area=600，正确效率应该是`600*floor/1200 = floor/2`。

这说明5.5.3中需要使用原始area而非base来计算效率，但文档未明确说明如何获取原始area。

**修复建议**: enrichConfigs中明确使用原始area（可通过duplexSource反推或config中增加area字段）

#### 问题4：两个相同base的config导致solveIntegerEquation搜索冗余

**位置**: ALGORITHM(2).md 子步骤4.3

**影响**: 不去重方案中，原生1200和转换1200作为两个独立config，base和unitCap完全相同。DFS搜索时这两个config的搜索分支完全等价，导致搜索空间翻倍但结果不变。

**建议**: 在solveIntegerEquation入口处检测并合并完全相同的config（base/unitCap/floor均相同），求解后再按比例拆分。或保持现状（性能影响极小，最多多1-2个重复配置）。

#### 问题6：MEMORY.md已知问题#8的验证

**位置**: MEMORY.md 已知错误第8条

**问题**: "密度约束未考虑固定产品基底"——在ALGORITHM(2).md 5.1节中`totals.base += fixedProductsBase`已包含固定产品基底，但solveIntegerEquation中的targetBaseLimit（4.4节`targetBaseBh * 1.05`）**不包含**固定产品基底。

这意味着Bh/Bl分配的基底约束只考虑了厂房部分，实际总基底=厂房基底+固定产品基底。如果固定产品基底很大，厂房基底即使不超1.05倍，总基底也可能超。

**建议**: 在4.4节中调整targetBaseLimit的计算，考虑固定产品基底：
```javascript
remainingTargetBaseBh = targetBaseBh - fixedProductsBase * (targetBaseBh / remainingBase)
bhResult = solveIntegerEquation(targetCapBh, bhConfigs, remainingTargetBaseBh * 1.05)
```
