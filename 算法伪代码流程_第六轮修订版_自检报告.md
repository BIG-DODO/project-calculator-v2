## 一、致命/高危问题（必须在代码实现前修复）

### 1. FL预处理固定配置数量未使用 FL 计算结果
**位置**：步骤三 3.3 节（第 471-512 行）

**问题描述**：
当前 FL 预处理虽然计算了 `FL = targetCap / a / 2`，但后续固定配置时直接对每种面积段固定 `count: 1`，完全没有使用 FL 值来决定固定栋数。

```
for area in productOptions[preprocessType].areas:
   base = area
   floors = maxFloor
   ...
   fixedConfigs.push({ ..., count: 1, isFixed: true })
```

**风险**：
- 如果 FL 很大（比如需要固定 5 栋），实际只固定 1 栋，导致后续 Bh/Bl 配置层数类型仍超过 2 种
- 如果 FL 很小（<1），按代码会 break，但此时可能仍有多种层数
- 固定 "每种面积段各 1 栋" 产品逻辑上不合理

**修复建议**：
- 按 FL 向下取整确定固定栋数：`fixCount = max(1, floor(FL))`
- 每种层数只固定 **最大面积段** 1 个配置（或按 FL 值固定多个），而不是每种面积段都固定
- 固定后从 `preprocessFloors` 中移除该层数

### 2. `mapOptimizedToCounts` 存在计数覆盖风险
**位置**：步骤五 5.4 节（第 1068-1094 行）

**问题描述**：
```
if bestIdx >= 0:
   counts[bestIdx] = opt.count
```
当多个 `optimized` 项映射到同一个现有配置时，后者会覆盖前者，导致丢失栋数。

**场景**：
- `manualDistribute` 中 `c` 经 10 取整后恰好等于 `a` 或 `b`
- 例如 a=540, b=800, 计算得 c=540，则 a 和 c 都映射到 540 配置，a 的 10 栋会被 c 的 7 栋覆盖

**修复建议**：
```
if bestIdx >= 0:
   counts[bestIdx] += opt.count   // 累加，不是赋值
```

### 3. `step5_outer_loop` 中 `S_type` 未在 enrich 后重新计算
**位置**：步骤五 5.2 节（第 898-925 行）

**问题描述**：
```
S_type = sum(allConfigs[i].base * counts[i])   // 基于 enrich 前的 counts
...
enrichedConfigs = enrichConfigs(...)
newCounts = solveIntegerEquation(...)
counts = newCounts.counts
...
if distCheck2.needsOptimization:
   optimized = manualDistribute(S_type, a, b, eff)   // 仍用旧 S_type
```

enrich 重新求解后，该类型的实际占地已经变化，但 `S_type` 没有更新，导致 `manualDistribute` 使用错误的总占地。

**修复建议**：
在调用 `manualDistribute` 前重新计算 `S_type`：
```
allConfigs = [...currentBhConfigs, ...currentBlConfigs]
typeIndices = allConfigs 中 id == type 的索引
S_type = sum(allConfigs[i].base * counts[i])
```

### 4. `mapOptimizedToCounts` 新配置归属逻辑不可靠
**位置**：步骤五 5.4 节（第 1088-1093 行）

**问题描述**：
```
if type == 'layer' 或 (type != 'layer' 且 blConfigs.length == 0):
   bhConfigs.push(newConfig)
else:
   blConfigs.push(newConfig)
```

该逻辑在两种情况下会出错：
1. 当 type='layer' 但实际配置在 blConfigs 中时（单一层数取消 Bh 后）
2. 当 type='split' 且 blConfigs 非空，但 split 实际属于 Bh 侧时

**风险**：新配置插入到错误的一侧，导致 `finalCounts` 索引错位或 Bh/Bl 分类混乱。

**修复建议**：
在步骤三中记录每种类型属于 Bh 还是 Bl，或者在 `step4` 后维护 `typeToSide` 映射表，按映射表插入。

### 5. `calc_dorm` 最终基底可能超出 600-1200 边界
**位置**：步骤二 2.3 节（第 415-448 行）

**问题描述**：
初始 base 被限制在 600-1200：
```
if singleArea < 600: base = 600
else if singleArea > 1200: base = 1200
else: base = round(singleArea / 100) * 100
```

但层数被压缩后，`exactBase = dormCap / floors` 可能 >1200 或 <600，此时 `finalBase = floor(exactBase / 20) * 20` 会超出边界。

**场景**：
- dormCap=10000, referenceHeight=15m, floors 被压缩到 2 层
- exactBase=5000, finalBase=5000（远超 1200）

**修复建议**：
在计算 `finalBase` 后增加边界检查：
```
finalBase = floor(exactBase / 20) * 20
finalBase = max(600, min(1200, finalBase))
unitArea = finalBase * floors
```

并评估此时 unitArea 与 dormCap 的偏差是否可接受。

## 二、中危问题（建议修复）

### 6. `step0_preprocess` 分层降级到 1 层时丢失
**位置**：第零步 0.1 节（第 198-212 行）

**问题描述**：
```
while totalH > Hl and currentFloor > 1:
   currentFloor--
   totalH = ...
if totalH <= Hl:
   if currentFloor >= 5: layerFloors.push(currentFloor)
   else if currentFloor <= 4: downgradedFloors.push(currentFloor)
```

如果限高极低（Hl < 9.3m），currentFloor 降到 1 后 totalH 仍 > Hl，while 退出，但 `totalH <= Hl` 不成立，该楼层既不进 layerFloors 也不进 downgradedFloors，**静默丢失**。

**修复建议**：
在降级后增加断言或错误提示：
```
if totalH > Hl:
   errors.push("分层厂房即使1层也无法满足限高要求")
```

### 7. `step3_factory_preprocess` 未检查 `remainingCap` 为负
**位置**：步骤三（第 585-586 行）

**问题描述**：
```
remainingBase = targetBase - fixedBase - usedBase
remainingCap = targetCap - fixedCap - usedCap
```

主流程只检查了 `remainingBase <= 0`，但如果固定产品计容面积已超过 `targetCap`，`remainingCap` 为负，后续算法仍会执行。

**修复建议**：
```
if remainingBase <= 0 或 remainingCap <= 0:
   adjustedResult = adjustFixedConfigs(...)
```

### 8. `enrichConfigs` 非轻钢类型排序无区分度
**位置**：步骤五 5.5 节（第 1127-1138 行）

**问题描述**：
```
targetEff = configs[0].unitCap / configs[0].base
...
if type == 'light-steel':
   eff = (area * 2 + 200) / area
else:
   eff = floor
return { area, diff: |eff - targetEff| }
```

对于分栋/分层，`eff = floor` 是常数，所有候选面积的 diff 相同，排序后取前 `maxAdd` 个完全取决于遍历顺序，可能不是最优选择。

**修复建议**：
对非轻钢类型，增加第二排序键：优先选择靠近现有面积段中点、或与目标基底（targetBase/N）接近的候选。

### 9. `manualDistribute` 对轻钢使用固定效率
**位置**：步骤五 5.3 节（第 1006-1052 行）

**问题描述**：
```
unitCap: base * eff
```
其中 `eff` 是调用方传入的该类型第一个配置的效率。对于轻钢，不同 base 的效率不同（(base*2+200)/base），用固定 eff 会导致 unitCap 计算不准确。

**修复建议**：
`manualDistribute` 接收配置数组而不是单一 eff，根据实际 base 计算 unitCap：
```
for opt in optimized:
   opt.unitCap = computeUnitCap(type, floor, opt.base)
```

### 10. `solveIntegerEquation` k=1 时初始 n 可能偏离最优

n = max(1, round(targetCap / c.unitCap))

如果目标值恰好处于两个整数中间，round 可能选错方向，但搜索范围 n-200 到 n+200 足够覆盖。这不是严重问题。

## 三、低危/注意事项

### 11. `fineTune` 密度超限时直接回退
**位置**：步骤五 5.6 节（第 1183-1185 行）

**问题描述**：
```
if |densityDiff| > targetBase * 0.05:
   if previousCounts: counts = previousCounts
   return { counts, totals }
```

一旦某次微调导致密度偏离 >5%，直接回退并返回。这可能导致最终密度仍在边界外，但外层循环的 break 条件也会因此不满足，最多 10 轮后退出。

**建议**：可接受，但需确保最终输出明确标注密度是否达标。

---

### 12. `checkDistribution` 只检查单个类型内部
**位置**：步骤五 5.2 节（第 955-1003 行）

**问题描述**：
分布均衡检查只检查同一类型内部各面积段的分布，不检查不同类型之间的比例。对于三种厂房类型同时存在的情况，可能需要额外检查。

**建议**：当前规则已足够覆盖用户提出的"某面积段超过其他 2 倍"问题，暂不需要扩展。
