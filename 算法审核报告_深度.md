## 二、严重问题（P0）—— 必须修复

### P0-1：步骤三 FL 预处理中 3.5 层分栋高度计算错误

**位置**：步骤三 3.3 节 / 3.4 节

**问题描述**：
当 FL 预处理固定分栋厂房的 `maxSplitFloor = 3.5` 时，代码直接使用通用公式计算高度：
```
totalH = 7.2 + 5.1 + 4.5 * (floors - 2) + 1.2
```
代入 `floors = 3.5`：
```
totalH = 7.2 + 5.1 + 4.5 * 1.5 + 1.2 = 20.25m
```

**但 3.5 层分栋的实际建筑高度应与 4 层相同（物理结构为 4 层，第 4 层只算半面积）**：
```
总层高 = 7.2 + 5.1 + 4.5 + 4.5 = 21.3m
建筑高度 = 21.3 + 1.2 = 22.5m
```

**影响**：
- 算法认为 3.5 层建筑高度 = 20.25m，实际 = 22.5m
- 当限高 Hl 在 20.25m ~ 22.5m 之间时（如 21m、22m），算法会**错误地允许 3.5 层**，而实际不可行
- 当限高 Hl 恰好为 22.5m 时，算法也会错误判断

**修复建议**：
在步骤三中使用 `computeTotalHeight()` 和 `computeUnitCap()` 工具函数，而非直接内联公式：
```javascript
totalH = computeTotalHeight('split', floors)  // 应返回 22.5
unitCap = computeUnitCap('split', floors, base)  // 应返回 base * 3.5
```

**自检报告状态**：未检出（自检报告仅关注了"FL 预处理 fixCount"和"三种厂房类型处理后清空剩余分层"，未覆盖此问题）

### P0-2：step0 与 `computeTotalHeight` 中 3.5 层建筑高度计算为 23.7m（应为 22.5m）

**位置**：第零步 0.2 节 / `computeTotalHeight` 函数

**问题描述**：
代码中 3.5 层的高度被硬编码为：
```javascript
if type == 'split' && floor == 3.5:
   return 22.5 + 1.2  // = 23.7
```

但 **22.5 已经是建筑高度（总层高 21.3 + 1.2）**，不应再加一次 1.2。正确值应为 **22.5**。

**验证**：
- 4 层分栋：`7.2 + 5.1 + 4.5*(4-2) + 1.2 = 7.2 + 5.1 + 9 + 1.2 = 22.5` ✓
- 3.5 层分栋：物理结构同 4 层，建筑高度应同为 22.5
- 代码返回：23.7（错误多了 1.2m）

**影响**：
- step0 中 3.5 层的限高判断偏差：当 Hl ∈ [22.5, 23.7) 时，3.5 层会被**错误地移除**（如 Hl = 23m 时，23.7 > 23，判定超限，但实际 22.5 < 23，应允许）
- 输出展示中高度字段错误

**修复建议**：
```javascript
if type == 'split' && floor == 3.5:
   return 22.5  // 22.5 已经是含女儿墙的建筑高度
```

**自检报告状态**：未检出

## 三、重要问题（P1）—— 建议修复

### P1-1：产业大厦"降标准层"逻辑完全缺失

**位置**：步骤一 `step1_tower` 函数

**问题描述**：
需求文档（MEMORY.md 中已确认的规则）明确要求：
> "产业大厦层高：首层 6.6m，标准层 4.5m；**超限时先降标准层到 4.2m**，再不行增加单层面积"

但算法中产业大厦限高超限的处理只有：
```javascript
if totalH > Hl:
   floors = max(3, floor((Hl - 6.6 - 4.5 - 1.2) / 4.5) + 2)
   totalH = 6.6 + 4.5 * (floors - 2) + 4.5 + 1.2
```

即**仅通过减少层数**来适应限高，完全没有实现"标准层从 4.5m 降至 4.2m"的策略。

**影响**：
- 某些场景下（如限高非常紧张时），产业大厦可能无法配置或配置层数过少
- 遗漏了需求文档中已确认的关键规则

**修复建议**：
增加标准层降级逻辑（先尝试降标准层到 4.2m，仍超限再减少层数）：
```javascript
function step1_tower(...) {
   // ... 初始计算 ...
   
   if totalH > Hl:
      // 尝试降标准层到 4.2m
      floors_with_reduced = max(3, floor((Hl - 6.6 - 4.5 - 1.2) / 4.2) + 2)
      totalH_reduced = 6.6 + 4.2 * (floors_with_reduced - 2) + 4.5 + 1.2
      
      if totalH_reduced <= Hl:
         floors = floors_with_reduced
         totalH = totalH_reduced
      else:
         // 仍超限，按原逻辑减少层数（标准层保持 4.5m）
         floors = max(3, floor((Hl - 6.6 - 4.5 - 1.2) / 4.5) + 2)
         totalH = 6.6 + 4.5 * (floors - 2) + 4.5 + 1.2
   
   // 继续后续取整逻辑...
}
```

> 注意：REQUIREMENTS_v1.2.md 正文未明确提及此规则，但 MEMORY.md 中已确认（"已确认的关键细节"第 11 条）。由于 MEMORY.md 是项目核心知识库，且自检报告声称"与 REQUIREMENTS_v1.2 一致"，此遗漏属于算法完整性缺陷。

### P1-2：步骤三 3.4 节"三种厂房类型处理"中高度计算未使用工具函数

**位置**：步骤三 3.4 节

**问题描述**：
在强制分配 1 栋最低层数、最小面积的分层，以及后续固定分栋最高层数时，代码直接内联了高度和计容公式：
```javascript
totalH = 7.2 + 5.1 + 4.5 * (floors - 2) + 1.2
unitCap = base * floors
```

**问题**：
1. 未调用 `computeTotalHeight()` 和 `computeUnitCap()`，导致 3.5 层等特殊场景处理不一致（见 P0-1）
2. 如果未来层高规则调整，需要修改多处代码，维护成本高

**修复建议**：
统一使用工具函数：
```javascript
totalH = computeTotalHeight(type, floors)
unitCap = computeUnitCap(type, floors, base)
```

## 四、中等问题（P2）—— 值得修复

### P2-1：FL 预处理 `FL < 1: break` 与需求语义存在歧义

**位置**：步骤三 3.3 节

**问题描述**：
需求文档规定：`fixCount = max(1, floor(FL))`。即即使 `FL < 1`（如 0.8），也固定 1 栋。

但算法中：
```javascript
if FL < 1: break
```

当 FL < 1 时，算法直接跳过固定，让后续处理。虽然后续有 `adjustFixedConfigs` 兜底，但语义上与需求不同。

**建议**：明确需求意图。如果需求确实要求 `fixCount = max(1, floor(FL))`，则应删除 `FL < 1: break` 条件，统一按 `fixCount = max(1, floor(FL))` 执行。

---

### P2-2：`enrichConfigs` 中 `targetBaseApprox` 估算使用 `c.count || 1` 不可靠

**位置**：`enrichConfigs` 函数

**问题描述**：
```javascript
estCount = sum(configs.map(c => c.count || 1))
targetBaseApprox = totalTargetBase / max(1, estCount)
```

`createConfigs` 创建的 config 没有 `count` 字段，所以 `c.count || 1` 恒为 1。`estCount` 实际上等于 `configs.length`，而非实际的预估栋数。这导致 `targetBaseApprox` 估算可能严重偏离实际目标基底。

**建议**：在创建 config 时传入预估的初始栋数（或从 `solveIntegerEquation` 的结果中获取），确保估算准确。

---

### P2-3：`solveIntegerEquation` fallback 返回全 0 可能违反"产品类型总栋数 ≥ 2"约束

**位置**：`solveIntegerEquation` 函数末尾

**问题描述**：
当 `maxN <= 0` 时：
```javascript
return { counts: configs.map(() => 0), totalCap: 0, totalBase: 0 }
```

但需求文档要求："数量下限：轻钢/分栋/分层 ≥ 2 栋（如勾选）"。返回全 0 违反了此约束。

**建议**：fallback 时至少返回 2 栋最小面积段配置，或在后续步骤中补充约束检查。

**MEMORY.md** 中同时提到：

算法当前实现与 REQUIREMENTS_v1.2 一致（固定值策略），但 MEMORY.md 中的 2:1 比例未体现。

**建议**：在 REQUIREMENTS_v1.2 的下一版本中明确统一配套用房规则，消除两份文档之间的歧义。

### P3-2：step0 中分层降至 1 层仍超限高时返回错误，但 1 层高度公式概念模糊

**位置**：第零步 0.1 节

**问题描述**：
当分层降至 1 层时：
```javascript
totalH = 7.2 + 5.1 + 4.5 * (1 - 2) + 1.2 = 7.2 + 5.1 - 4.5 + 1.2 = 9.0
```

但 1 层建筑不应有"二层"（5.1m）和"标准层"（-4.5m）。虽然数学上算出来是 9.0m，但概念上 1 层建筑的高度应该是首层 + 1.2 = 8.4m（如果只有 1 层）或者首层 + 顶层 + 1.2 = 7.2 + 4.5 + 1.2 = 12.9m（如果按配套楼逻辑）。

**建议**：明确 1 层建筑的高度定义。当前逻辑在 step0 中已返回错误，实际影响较小。

---

### P3-3：`solveIntegerEquation` 多配置 DFS 中 `1.5` 倍硬截断可能跳过最优解

**位置**：`solveIntegerEquation` 多配置 DFS 分支

```javascript
if hasBaseLimit && currentBase > targetBaseLimit * 1.5: return
if currentCap > targetCap * 1.5: return
```

1.5 倍截断是经验值，在极端场景下可能跳过理论上更优的解。当前测试场景未触发此问题。
