#### 问题1：enrichConfigs中轻钢效率计算与实际效率不一致

**位置**: 算法伪代码流程_完整版.md 第947行

**问题描述**:
```javascript
// 第947行
filteredCandidates = [...candidates]
   .filter(area => !existingAreas.has(area))
   .map(area => {
      eff = type == 'light-steel' ? 2.1 : floor   // ← 轻钢固定为2.1
      return { area, diff: |eff - targetEff| }
   })
```

轻钢的实际效率随base变化：`unitCap/base = (base * 2 + 200)/base = 2 + 200/base`

| base | 实际效率 | 固定值2.1 | 偏差 |
|------|---------|----------|------|
| 600 | 2.333 | 2.1 | 0.233 |
| 800 | 2.250 | 2.1 | 0.150 |
| 1000 | 2.200 | 2.1 | 0.100 |
| 1200 | 2.167 | 2.1 | 0.067 |

而targetEff基于configs[0]计算（第942行）：
```javascript
targetEff = configs[0].unitCap / configs[0].base  // 实际效率
```

**矛盾**: 排序用的eff=2.1（固定值），但targetEff是实际效率（如2.333），两者相差0.233。这会导致新增面积段的排序偏离真实效率接近程度。

**修复建议**:
```javascript
.map(area => {
   if type == 'light-steel':
      eff = (area * 2 + 200) / area  // 使用实际效率
   else:
      eff = floor
   return { area, diff: |eff - targetEff| }
})
```

#### 问题5：remainingBase为负时的处理

**位置**: 算法伪代码流程_完整版.md 第500行

**问题描述**:
```javascript
remainingBase = targetBase - fixedBase - usedBase
```

如果固定产品（产业大厦+配套楼+配套宿舍）的基底 + 厂房固定配置（FL预处理）的基底 > targetBase，remainingBase为负。

后续solveIntegerEquation中（第612行）：
```javascript
hasBaseLimit = targetBaseLimit != null 且 targetBaseLimit > 0
```

如果remainingBase <= 0，hasBaseLimit=false，求解时无基底约束。这可能导致：
- 厂房部分使用最小面积段尽可能满足计容
- 总基底远超targetBase
- 外层循环检测到密度超标后回退

**建议**: 在步骤三末尾增加负值检查：
```javascript
if remainingBase <= 0:
   // 固定产品已超过密度限制，需要调整
   // 优先减少FL预处理中的固定配置
   // 或返回错误提示
```
